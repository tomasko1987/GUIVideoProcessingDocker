1. Dockerfile

Dockerfile je súbor s inštrukciami, ako zostaviť Docker image. Nachádza sa v koreni projektu: c:\Apps\Docker\GUIVideoProcessing.Web\Dockerfile

Má dve fázy (multi-stage build):

Fáza 1 – Build (SDK)

Použije oficiálny .NET 9 SDK image, skopíruje zdrojové súbory a zostaví aplikáciu.

Fáza 2 – Runtime (Ubuntu 22.04)

Výsledný image je postavený na Ubuntu 22.04 (nie na Debian Bookworm). Dôvod: knižnica libOpenCvSharpExtern.so (OpenCV natívny bridge) vyžaduje konkrétne verzie systémových knižníc (libjpeg-turbo8, libavcodec58 atď.), ktoré sú dostupné práve v Ubuntu 22.04.

Dockerfile 
# ── Fáza 1: Build ─────────────────────────────────────────
FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build
WORKDIR /src
COPY *.csproj .
RUN dotnet restore
COPY . .
RUN dotnet publish -c Release -o /app/publish

# ── Fáza 2: Runtime ───────────────────────────────────────
FROM ubuntu:22.04 AS runtime
WORKDIR /app

ENV DEBIAN_FRONTEND=noninteractive

# Inštalácia ASP.NET Core 9 runtime
RUN apt-get update \
    && apt-get install -y wget ca-certificates \
    && wget https://packages.microsoft.com/config/ubuntu/22.04/packages-microsoft-prod.deb \
         -O packages-microsoft-prod.deb \
    && dpkg -i packages-microsoft-prod.deb \
    && rm packages-microsoft-prod.deb \
    && apt-get update \
    && apt-get install -y aspnetcore-runtime-9.0 \
    && rm -rf /var/lib/apt/lists/*

# Natívne knižnice pre OpenCV
RUN apt-get update && apt-get install -y \
    libgdiplus \
    libx11-6 \
    libxext6 \
    libglib2.0-0 \
    libgomp1 \
    libjpeg-turbo8 \
    libavcodec58 \
    libavformat58 \
    libavutil56 \
    libswscale5 \
    libtesseract4 \
    libgtk2.0-0 \
    libopenexr25 \
    && rm -rf /var/lib/apt/lists/*

COPY --from=build /app/publish .

RUN adduser --disabled-password --gecos "" appuser \
    && chown -R appuser /app
USER appuser

EXPOSE 5250
ENTRYPOINT ["dotnet", "GUIVideoProcessing.Web.dll"]
Prečo Ubuntu 22.04 a nie mcr.microsoft.com/dotnet/aspnet:9.0?
Oficiálny ASP.NET Core runtime image je postavený na Debian Bookworm, ktorý má iné (novšie) verzie knižníc ako tie, pre ktoré bol skompilovaný OpenCvSharp4. Ubuntu 22.04 obsahuje správne verzie (libjpeg-turbo8, libavcodec58 atď.) a OpenCV sa v ňom načíta bez chýb.

2. Build a prenos na VM

Všetky príkazy v tejto kapitole sa spúšťajú vo Windows PowerShell, nie v SSH relácii na Ubuntu VM.

Krok 1 – Zbaliť projekt

Adresáre bin/, obj/, .vs/ a wwwroot/lib/ sa vynechávajú — sú buď zbytočné (Docker zostaví vlastné binárky) alebo veľké (npm knižnice).

Windows PowerShell 
Get-ChildItem "c:\Apps\Docker\GUIVideoProcessing.Web" `
    -Exclude "bin","obj",".vs","wwwroot\lib" |
  Compress-Archive -DestinationPath "c:\Apps\Docker\project.zip" -Force
Parameter -Force automaticky prepíše existujúci project.zip. Nie je potrebné ho pred tým manuálne mazať.

Krok 2 – Preniesť ZIP na VM

Windows PowerShell 
scp "c:\Apps\Docker\project.zip" tom@192.168.50.103:/opt/guivideoprocessing/

Krok 3 – Pripojiť sa na VM cez SSH

Windows PowerShell 
ssh tom@192.168.50.103

Krok 4 – Zmazať starý projekt a rozbaliť nový

Ubuntu VM (SSH) 
sudo rm -rf /opt/guivideoprocessing/project
unzip /opt/guivideoprocessing/project.zip -d /opt/guivideoprocessing/project

Krok 5 – Zostaviť Docker image

Build prebieha priamo na Ubuntu VM — stiahne všetky knižnice a zostaví image.

Ubuntu VM (SSH) 
cd /opt/guivideoprocessing/project
docker build -t guivideoprocessing:latest .
Build trvá 3–10 minút (závisí od rýchlosti internetu VM). Na konci musíš vidieť:
Successfully built <id>
Successfully tagged guivideoprocessing:latest

3. Spustenie kontajnera

Zastaviť a odstrániť starý kontajner

Ubuntu VM (SSH) 
docker stop guivideoprocessing && docker rm guivideoprocessing

Spustiť nový kontajner

Ubuntu VM (SSH) 
docker run -d \
  --name guivideoprocessing \
  -p 5250:5250 \
  --restart unless-stopped \
  -v /opt/guivideoprocessing/logs:/app/logs \
  -v /opt/guivideoprocessing/appsettings.json:/app/appsettings.json \
  guivideoprocessing:latest

5. Logy

Logy sú zapisované na dva miesta súčasne: do konzoly kontajnera a do súborov v priečinku /opt/guivideoprocessing/logs/ na Ubuntu VM.

Zobraziť logy v termináli

6. Časté problémy

Kamera sa nepripája (Connection timeout)

V logoch sa opakuje MJPEG: Connection timeout after 15s. Will retry.

  • Skontroluj či je ESP32-CAM fyzicky zapnutá
  • Skontroluj či je pripojená k WiFi sieti
  • Overiť dostupnosť: ping 192.168.50.96 z Ubuntu VM
  • Skontrolovať URL v appsettings.json → Camera:Urls

appsettings.json – Read-only file system

Chyba Read-only file system: '/app/appsettings.json' sa objaví ak je kontajner spustený s :ro na konci mount parametra.

Oprava – spusti kontajner bez :ro:

-v /opt/guivideoprocessing/appsettings.json:/app/appsettings.json

Zmeny nastavení sa neprejavia bez reštartu

Na Linuxe v Docker kontajneri s bind-mountovaným súborom FileSystemWatcher nespoľahlivo detekuje zmeny súboru. Aplikácia volá IConfigurationRoot.Reload() po každom zápise do appsettings.json — zmeny sa prejavia okamžite bez reštartu.

Číslice sa nezobrazujú (/left alebo /right vracia prázdny obrázok)

  • Pipeline nespracúva framy → kamera nie je dostupná
  • Skontroluj logy: docker logs guivideoprocessing 2>&1 | grep -E "ONNX|SevenSegment"
  • ROI je mimo obrazu → uprav ROIX, ROIY v GUI alebo appsettings.json

Port 5250 nie je dostupný z prehliadača

  • Skontroluj či kontajner beží: docker ps
  • Skontroluj Kestrel URL v appsettings.json: musí byť http://0.0.0.0:5250
  • Skontroluj firewall na Ubuntu VM: port 5250 musí byť otvorený

docker build zlyhá – knižnica not found

Po úspešnom builde spusti v kontajneri: 

docker exec guivideoprocessing ldd /app/runtimes/linux-x64/native/libOpenCvSharpExtern.so | grep "not found"

Ak vypíše nejakú knižnicu, je potrebné ju pridať do Dockerfile do sekcie apt-get install a znova buildn­úť.

7. Code Update

Ak zmeníš zdrojový kód aplikácie (C#, Razor Pages, JS, CSS…), je potrebné zostaviť nový Docker image a nasadiť ho. Zmena kódu si vždy vyžaduje rebuild. Jedinou výnimkou je appsettings.json — ten sa mení priamo na VM bez rebuildu.

Čo nevyžaduje rebuild: zmena appsettings.json (ROI, algoritmus, InfluxDB…)
Čo vyžaduje rebuild: akákoľvek zmena v .cs, .cshtml, .js, .css, Dockerfile alebo .csproj súboroch.

8. Docker vs Win – Porovnanie projektov

Existujú dve verzie projektu:

  • Windows verzia: C:\Apps\GUIVideoProcessing.Web — spúšťa sa priamo ako aplikácia na Windows 11
  • Docker verzia: C:\Apps\Docker\GUIVideoProcessing.Web — nasadzuje sa do Docker kontajnera na Ubuntu VM

Väčšina kódu je identická. Rozdiely sú minimálne a cielené — každá zmena mala konkrétny dôvod. Nižšie sú popísané všetky rozdiely súbor po súbore.

1. GUIVideoProcessing.Web.csproj — NuGet balíky

Windows verzia

Windows projekt 
<PackageReference Include="OpenCvSharp4.Windows" Version="4.13.0.20260226" />

Docker verzia

Docker projekt 
<PackageReference Include="OpenCvSharp4.Windows"                   Version="4.13.0.20260302" />
<PackageReference Include="OpenCvSharp4.official.runtime.linux-x64" Version="4.13.0.20260302" />

Vysvetlenie

OpenCvSharp4.Windows obsahuje natívne DLL knižnice pre Windows (OpenCvSharpExtern.dll). Na Linuxe tieto DLL nefungujú.

Pre Docker kontajner (Linux) je preto pridaný balík OpenCvSharp4.official.runtime.linux-x64, ktorý obsahuje natívnu knižnicu libOpenCvSharpExtern.so pre Linux x64.

Oba balíky môžu existovať v jednom projekte súčasne — .NET si pri builde automaticky vyberie správnu platformu (Windows DLL alebo Linux .so).

Verzia balíka bola zároveň aktualizovaná z 20260226 na 20260302 kvôli kompatibilite s novšou verziou natívnych knižníc na Ubuntu 22.04.

2. Program.cs — HTTPS presmerovanie

Windows verzia

Windows projekt 
// Presmeruj HTTP → HTTPS automaticky
app.UseHttpsRedirection();

Docker verzia

Docker projekt 
// UseHttpsRedirection() — vypnuté, aplikácia beží v Docker len cez HTTP

Vysvetlenie

Na Windowse ASP.NET Core automaticky presmeruje HTTP požiadavky na HTTPS. To je žiaduce pri lokálnom spustení, kde má aplikácia vlastný HTTPS certifikát.

V Docker kontajneri táto funkcia spôsobovala chybu — kontajner nemal nakonfigurovaný HTTPS certifikát, takže každá HTTP požiadavka skončila presmerovaním na HTTPS, ktoré nefungovalo. Výsledok: prázdna stránka v prehliadači.

V Docker prostredí sa HTTPS rieši na úrovni reverse proxy (napr. nginx) pred kontajnerom. Samotná aplikácia komunikuje cez čisté HTTP.

3. appsettings.json — Kestrel URL binding

Windows verzia

Windows projekt 
"Kestrel": {
  "Endpoints": {
    "Http": {
      "Url": "http://localhost:5250"
    }
  }
}

Docker verzia

Docker projekt 
"Kestrel": {
  "Endpoints": {
    "Http": {
      "Url": "http://0.0.0.0:5250"
    }
  }
}

Vysvetlenie

localhost znamená, že aplikácia počúva len na loopback rozhraní (127.0.0.1) — prístupná iba z toho istého počítača. Na Windowse je to správne správanie pre lokálnu aplikáciu.

V Docker kontajneri má každý kontajner vlastnú sieťovú kartu s vlastným localhost. Keby aplikácia počúvala na localhost, požiadavky z hostiteľa (Ubuntu VM) by sa k nej nikdy nedostali.

0.0.0.0 znamená počúvaj na všetkých sieťových rozhraniach — vrátane toho, cez ktoré Docker mapuje port 5250 z hostiteľa do kontajnera.

4. RoiController.cs — okamžitý reload konfigurácie

Windows verzia

Windows projekt 
private readonly IConfiguration _config;

public RoiController(IConfiguration config, IWebHostEnvironment env, ILogger<RoiController> logger)
{
    _config = config;
    _env    = env;
    _logger = logger;
}

// Po zápise do súboru:
System.IO.File.WriteAllText(filePath, json);
// (žiadny reload — spolieha sa na reloadOnChange)

Docker verzia

Docker projekt 
private readonly IConfiguration _config;
private readonly IConfigurationRoot _configRoot;   // ← nové

public RoiController(IConfiguration config, IWebHostEnvironment env, ILogger<RoiController> logger)
{
    _config     = config;
    _configRoot = (IConfigurationRoot)config;       // ← nové
    _env        = env;
    _logger     = logger;
}

// Po zápise do súboru:
System.IO.File.WriteAllText(filePath, json);
_configRoot.Reload();                               // ← nové

_configRoot.Reload() je pridané v metódach: MoveRoi a ResizeRoi.

Vysvetlenie

ASP.NET Core štandardne sleduje zmeny v appsettings.json cez FileSystemWatcher (reloadOnChange=true) a automaticky obnovuje IConfiguration v pamäti.

Na Windowse toto funguje spoľahlivo. Na Linuxe v Docker kontajneri s bind-mountovaným súborom však FileSystemWatcher (interne používa inotify) nespoľahlivo detekuje zmeny súboru zapísaného cez File.WriteAllText(). Výsledok: zmena ROI v GUI sa prejavila až po reštarte kontajnera.

Riešenie: po každom File.WriteAllText() sa zavolá _configRoot.Reload(), ktorý okamžite znovu načíta celý appsettings.json priamo do pamäte — bez čakania na FileSystemWatcher.

5. PipelineController.cs — okamžitý reload konfigurácie

Rovnaká zmena ako v RoiController.cs. Pridaný _configRoot a _configRoot.Reload() po každom zápise do appsettings.json.

Týka sa všetkých SET metód:

  • SetBilateral
  • SetClahe
  • SetAlgorithm
  • SetThreshold
  • SetMorphology
  • SetSplit
Na Windowse táto zmena nemá žiadny negatívny efekt — Reload() len zopakuje to, čo by sa aj tak stalo automaticky o chvíľu neskôr.
